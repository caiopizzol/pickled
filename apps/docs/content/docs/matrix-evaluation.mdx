---
title: Matrix evaluation
description: Test what agents say about your product across interfaces, sources, and tool paths in one config.
---

A matrix scenario expands into one cell per `(interface × source × toolset)` tuple. Each cell runs independently and scores against its own contract, so the report shows where the agent does well and where it does not. The design lives in `proposals/matrix-evaluation.md` in the repo.

## Declare a matrix

```yaml
scenarios:
  - name: Install
    prompt: How do I install my-product?
    matrix:
      interfaces: [quick, codex]
      sources: [readme, llms]
      toolsets: [none, web]
    expected:
      includes: ["bunx my-product"]
```

That scenario produces eight cells. Each cell label is `[interface · source · toolset]`.

`matrix.sources` accepts a reserved value `none` alongside registered source ids. A `none` cell gets no source content and no discovery hint. Combined with `toolset: none` it measures model prior (no source, no tools); combined with `toolset: web` or `toolset: mcp` it measures open discovery where the agent must find the answer from scratch with its configured tools.

## Three toolset shapes

- **`none`** is the deterministic baseline. Pickled injects the cell's active source content into the agent's prompt. Citation contract applies if `requiredSources` is declared.
- **`web`** runs on Claude Code, the Anthropic API target, and the OpenAI API target. On Claude Code the cell scopes built-ins to `[WebSearch, WebFetch]` so the agent cannot fall back to filesystem; on the Anthropic API target the cell passes the server-side `web_search` tool (`web_search_20250305`) to `messages.create`; on the OpenAI API target the cell passes the `web_search` tool to `responses.create`. Source is NOT injected on any path; the cell scores on `expected.includes`/`excludes` plus tool-use provenance.
- **`mcp`** runs on Claude Code and on the OpenAI API target. On Claude Code the configured MCP servers are passed to the Agent SDK natively (`tools: []` disables built-ins, `allowedTools: ["mcp__<server>__*"]` auto-permits); on the OpenAI API target each server becomes a hosted-MCP tool entry on `responses.create` (`type: "mcp"`, `server_label` = map key, `server_url` = `url`, `headers` forwarded). Provenance accepts any invocation of `mcp__<server>__*` regardless of provider.

See [Toolsets](/toolsets) for the full shape of each profile.

## Scoring

Three signals compose per cell:

1. **Trap firing** is a universal veto. Any matched trap forces `NO` with confidence `0`.
2. **Citation** (in `none` cells with `requiredSources`) or **expected substrings** (in any cell with `expected.includes`/`excludes`) compose with worst-verdict + average-confidence.
3. **Tool-use provenance** (in non-`none` cells): a cell that does not invoke at least one of the configured tools is hard-vetoed to `NO`, regardless of substance. An answer pulled from model prior knowledge cannot testify to the tool path the cell is meant to test.

## Filter cells in CI

The CLI accepts cell filters so a GitHub Actions matrix job can run exactly one cell at a time:

```bash
pickled check . \
  --interface quick \
  --source readme \
  --toolset web \
  --scenario "Install"
```

See [GitHub Actions](/github-actions) for a full matrix workflow.

## Plan, cap, and sample

The matrix can grow expensive fast as you add interfaces and tool paths. Four flags help you stay in control without hand-editing axes:

- `--plan` (dry run): expand the matrix, apply filters and sampling, print the cell list, and exit. Zero model calls. Useful before a paid run to see what would happen.
- `--max-cells N`: hard cap on cells after filters and sampling. Exits non-zero before any model call if exceeded. Forces you to add filters or sample rather than silently spending on an oversized matrix.
- `--sample N`: deterministic per-scenario sample. Picks up to N cells per matrix scenario; single-cell (non-matrix) scenarios always run as deterministic anchors.
- `--seed VALUE`: seed for `--sample`. Defaults to the literal string `default` so reruns without `--seed` produce the same sample.

```bash
# See what would run, no model calls
pickled check . --plan

# Refuse to run a matrix larger than 10 cells
pickled check . --max-cells 10

# Sample 2 cells per scenario, reproducibly
pickled check . --sample 2 --seed nightly-2026-05-24
```

The receipt header (text and JSON) records `expandedCells`, `selectedCells`, and `seed` so a reviewer can see what was skipped and rerun the same sample.

## Readiness diagnostics

After a run completes, pickled walks the per-cell receipts and emits a structured `readiness.diagnostics` summary. Each diagnostic is a pattern over fields already in the report (no model is asked to interpret receipts). v1 surfaces five patterns:

- `grouped_check_pass` - a cell satisfied every entry across every declared `expected.{symbols,paths,options,constraints}` group; full readiness signal.
- `source_comparison` - a named source answered while `source: none` (model prior) did not on the same `(interface, toolset)` axis; the docs are doing the comprehension work.
- `toolset_comparison` - a tool-enabled cell answered while `toolset: none` did not on the same `(interface, source)` axis; the agent needed live discovery to reach the answer.
- `interface_comparison` - at least one interface scored YES and another scored worse on the same `(source, toolset)`; provider-specific gap.
- `trap_attribution` - one or more declared traps fired; hoisted to a scenario-level diagnostic so the reporter can show the stale-content pipeline is working at a glance.

The terminal output prints a `Readiness diagnostics` block when at least one pattern applied; otherwise it stays silent. The JSON receipt carries the same diagnostics under `readiness.diagnostics` with structured `pattern` / `scenario` / `cells` fields so downstream tooling (CI status, dashboards) can consume them.
